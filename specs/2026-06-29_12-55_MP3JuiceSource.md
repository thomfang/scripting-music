## 1. 核心目标（Loop Anchor）

让用户在「在线搜索 / 下载」时，可以**选择服务源**：现有聚合源（001-co，多 provider）或新增的 **MP3Juice（mp3juice3.ninja + savetube.vip，YouTube 音视频）**。新增源必须与现有 `MusicData` / `Music` 数据结构对齐，复用既有播放、下载、收藏、入库链路，**尽量不破坏现有调用点**。

## 2. 现有架构事实（已读代码确认）

### 2.1 数据结构
- 搜索结果：`MusicData { id, title, provider, artist?, cover?, album?, duration? }`（`class/music.ts`）
- 入库/播放：`Music { id, title, artist, album, duration, cover_url?, audio_url?, provider?, source_id?, is_downloaded, ... }`（`class/database.ts`）
- `provider` 当前是**有限白名单**：`MusicProvider = "livepoo"|"migu"|"qqmp3"|"qq"|"bugu"|"gequhai"|"gequbao"`，`SUPPORTED_PROVIDERS` + `isSupportedProvider()`。

### 2.2 单例 `music`（`class/music.ts`）——一切的接缝
- `music.search(query, provider="all") → {items: MusicData[]}`：GET `001-co.vercel.app/api/search`。
- `music.getAudioUrl(id, provider) → string`：**同步**拼一个 GET URL，直接当音频直链。
- `music.download(id, provider) → bytes`（实际下载走 downloader，不走这个）。

### 2.3 调用点（grep 已确认）
- `music.search`：`page/search/index.tsx:127`、`class/fetch_downloader.ts:128`（资源修复重搜）、`resource_repair_match.ts`（注释）。
- `getAudioUrl`：`search/index.tsx:156`、`search_result_card.tsx:35`、`player.ts:277/295`、`fetch_downloader.ts:90/161`、`resource_repair.tsx:203/216`。
- `SUPPORTED_PROVIDERS`/`isSupportedProvider`：`diag_db.ts`、`fetch_downloader.normalizeProvider()`。

### 2.4 下载链路（`class/fetch_downloader.ts`）
- `downloadMusic(info)` → `resolveDownloadInfo()`（探活 `audio_url`，失效则 `findReplacementSource()` 跨 provider 重搜）→ `performDownload()`：**单次 `fetch(audio_url)` + content-length + ReadableStream reader 流式**下载 → ID3 写入封面 → 入库。
- 关键约束：**下载器假定 `audio_url` 是一个可直接 GET 的最终直链**，且 `getAudioUrl` 是同步的。

### 2.5 播放链路（`class/player.ts:259-308`）
- `is_downloaded` → 本地路径；否则用 `music.audio_url`；为空且有 `provider` 时**同步** `getAudioUrl(source_id??id, provider)` → `AVPlayer.setSource(url)`。

### 2.6 搜索 UI（`page/search/index.tsx`）
- 已有 segmented `Picker`：`mode = online|local`。源选择可加在这里（再加一个 Picker / Menu）。

## 3. MP3Juice 流程事实（已真机逆向 + 实跑验证，Radiohead Creep 通过）

1. 搜索：`POST mp3juice3.ninja/api/yt-data` body `{query}` → `{items:[{id,title,duration:"3:57",thumbnail,url}]}`（YouTube videoId）。
2. 下载（savetube.vip）：
   - `GET media.savetube.vip/api/random-cdn` → `{cdn}`
   - `POST https://<cdn>/v2/info` body `{url:youtubeUrl}` → `{data: <base64 AES-128-CBC 密文>}`
   - 解密（IV=前16字节，key 固定 `C5D58EF67A7584E4A29F6C35BBC4EB12`）→ `{key,title,audio_formats,video_formats,...}`
   - **MP3**：`POST https://<cdn>/download` body `{downloadType:"audio",quality:128,key}` → `{data:{downloadUrl}}`（最终 .mp3 直链）
   - **MP4**：解密 info 的 `video_formats[i].url` 即 googlevideo 直链。
- 已逆向函数：workspace `mp3juice.ts`（`search` / `getDownloadUrl`）。

### 3.1 与现有架构的关键阻抗失配（必须解决的取舍点）
| 维度 | 现有源（001-co） | MP3Juice |
|---|---|---|
| 取直链 | **同步** `getAudioUrl(id,provider)` 一步拼 URL | **异步多步**：random-cdn→/v2/info→AES 解密→/download |
| 直链时效 | 长期稳定 | **短时签名**（googlevideo expire，mp3 直链也可能过期）→ 不应久存 DB |
| duration | number(秒) | 字符串 `"3:57"`，需解析为秒 |
| cover | `cover` | `thumbnail`（i.ytimg.com） |
| id 语义 | provider 侧 id | YouTube videoId（同时是 `source_id`） |

**结论**：`getAudioUrl` 的「同步返回最终直链」假设对 MP3Juice 不成立。必须引入**异步直链解析**接缝。

## 4. 方案权衡

### 方案 A：最小侵入 — 在 `music` 单例上加异步解析 + 源开关（推荐）
- 把「源」抽象成内部分支，对外仍是 `music.search()` + 新增 `music.resolveAudioUrl()`（async）。
- `MusicProvider` 扩展加 `"mp3juice"`；`provider` 字段复用承载源标识（mp3juice 的 item.provider="mp3juice"）。
- 新增 `class/sources/` ：`source.ts`（接口）、`source_001co.ts`（包装现有逻辑）、`source_mp3juice.ts`（逆向逻辑搬入）。
- `music.ts` 变成 dispatcher：按 `provider` 路由到对应 source。
- 新增 **`resolveAudioUrl(info): Promise<string>`**：001-co 直接返回 `getAudioUrl()`；mp3juice 走多步解析。
- 改造调用点：
  - `fetch_downloader.resolveDownloadInfo()`：用 `await music.resolveAudioUrl(info)` 替代 `getAudioUrl()`（已是 async，天然契合）。
  - `player.ts`：把两处同步 `getAudioUrl` 改成 `await music.resolveAudioUrl(...)`（player 该函数已是 async）。
  - `search_result_card.handlePlay`：播放前 `await` 解析（已 async）。
  - `search/index.tsx`：`doOnlineSearch` 传入当前选中源；新增源选择 Picker。
- `getAudioUrl()` **保留**（001-co 同步快路径 + 历史兼容），新增 `resolveAudioUrl()` 作为统一异步入口（内部对 001-co 调 getAudioUrl）。

优点：改动集中、风险可控、向后兼容、与既有 async 下载/播放链路契合。
缺点：`MusicProvider` 联合类型 + 白名单要同步扩展（diag/repair 需识别 mp3juice 或显式排除）。

### 方案 B：完整 SourceRegistry 重构
- 抽象 `MusicSource` 接口 + registry，所有源平等注册，UI 动态列源。
- 优点：未来加源零成本、最干净。
- 缺点：改动面大、回归风险高、超出「加一个源」的当前诉求 → 过度工程。

### 方案 C：把 mp3juice 完全独立成第二套 UI/链路
- 缺点：重复播放/下载/入库逻辑，违背「对齐数据结构、复用链路」目标。直接否决。

**推荐：方案 A**（必要时 source 接口设计成可平滑升级到 B 的形状）。

## 5. 方案 A 详细设计

### 5.1 新增/修改文件
- 新增 `class/sources/source.ts`：
  ```ts
  export interface MusicSource {
    id: MusicProvider            // "mp3juice" 等
    label: string                // UI 显示名
    search(query: string): Promise<MusicData[]>
    /** 解析可直接 GET 的最终音频直链（mp3）。可异步多步。 */
    resolveAudioUrl(info: ResolveInput): Promise<string>
    /** 可选：视频直链 */
    resolveVideoUrl?(info: ResolveInput): Promise<string>
  }
  ```
- 新增 `class/sources/source_001co.ts`：包装现有 search + getAudioUrl（resolveAudioUrl 同步返回）。
- 新增 `class/sources/source_mp3juice.ts`：搬入逆向 search / random-cdn / AES 解密 / download；`search()` 把 `duration "3:57"→秒`、`thumbnail→cover`、`provider:"mp3juice"`、`id=videoId`。
- 改 `class/music.ts`：
  - `MusicProvider` 增加 `"mp3juice"`；`SUPPORTED_PROVIDERS` 增加；导出 `PROVIDER_SOURCES`/源列表。
  - `music.search(query, source?)` 路由；新增 `music.resolveAudioUrl(info)`、`music.listSources()`。
  - 保留 `getAudioUrl()` 不动（001-co 内部用）。
- 改调用点：`fetch_downloader`（resolve 用异步）、`player.ts`（两处）、`search_result_card`（handlePlay/handleDownload）、`search/index.tsx`（源 Picker + 透传源）。

### 5.2 数据结构对齐映射（MP3Juice → MusicData/Music）
```
yt item            MusicData/Music
-----------------  --------------------------------
id (videoId)    →  id, source_id
title           →  title
"Radiohead"?    →  artist（yt 无独立 artist，title 整体进 title，artist 留空/“未知艺术家”）
thumbnail       →  cover / cover_url
duration "3:57" →  duration(=237, parseDuration)
url             →  （解析用，不直接入库 audio_url）
provider        →  "mp3juice"
audio_url       →  入库时留空；播放/下载时 resolveAudioUrl 动态解析（短时直链不久存）
```

### 5.3 短时直链处理
- mp3juice 的 audio_url **不写入 DB**（或写入后视为可失效）；播放/下载时实时 `resolveAudioUrl()`。
- 下载器 `resolveDownloadInfo` 已有「探活失败→重搜」机制；mp3juice 直接每次重解析即可，天然适配。
- 资源修复 `findReplacementSource` 跨 provider 重搜：mp3juice item 也能作为候选（需 resolveAudioUrl 探活）。

### 5.4 白名单/诊断影响
- `isSupportedProvider("mp3juice")` 返回 true（加入白名单），避免被资源修复页误判为「无效 provider」。
- `fetch_downloader.normalizeProvider` / `retryProviders`：mp3juice 不应被 `normalizeProvider` 回退成 migu；需在分支里优先按 source 路由。

### 5.5 UI
- `search/index.tsx` 在「搜索模式」Picker 旁/下再加一个**源选择**（Menu 或 segmented Picker：`聚合源 | MP3Juice`），online 模式生效；切源重新搜索。
- mp3juice 结果卡片复用 `SearchResultCard`（字段已对齐）。

## 11. Phase 3（2026-06-29 14:15）——修复下载（新版 fetch / FileManager 二进制 API）

### 问题
用户：搜索 OK、播放 OK，但点击下载失败；另问 mp3juice 是否有更多页。

### 分页结论
`mp3juice3.ninja/api/yt-data` **不支持分页**：page/offset/continuation 参数均被忽略，固定返回 20 条（YouTube 搜索首页）。接口响应只有 `items`，无游标/nextPageToken。无法从该接口取更多页。

### 下载失败根因（两个新版 native API 变更，均实测）
1. **fetch 流类型变了**：新版 `response.body` 是标准 `ReadableStream<Uint8Array>`（chunk 无 `.toUint8Array()`）；老的 `Data` 流挪到 `response.dataStream`。旧代码对 body chunk 调 `.toUint8Array()` 在新版报 `self type check failed for Objective-C instance method`。
2. **`FileManager.writeAsBytes`（async）整体失效**：报 `invalid "Data" argument`。精确隔离结论（iOS 26.5 实测）：
   - 连最简单的 `new Uint8Array([1,2,3,4,5])`（byteOffset=0、buffer长==length、非 shared）也失败 → **不是 sub-view 边缘问题，是该 binding 整体损坏**。
   - **`writeAsBytesSync`（sync）✅ 成功**；**`writeAsData`（async）✅ 成功**且内容往返正确。
   - → bug 精确限在 **async `writeAsBytes` 这一个绑定**（sync 版和 writeAsData 都正常）。应反馈开发者。

### Change Log
- 改 `class/fetch_downloader.ts` 流读取：`const stream = (resp as any).dataStream ?? resp.body`，chunk 归一化 `typeof v.toUint8Array==="function" ? v.toUint8Array() : (v as Uint8Array)`。
- 新增 `class/write_compat.ts` `writeBytesCompat(path, bytes)`：优先 `Data.fromUint8Array + writeAsData`，失败/无该 API 回退 `writeAsBytes`（老版）。
- 3 个写入调用点改用 `writeBytesCompat`：`file_manager.saveAudio/saveCover`、`fetch_downloader` 最终写入。

### Validation
- `scripting-ts run` 端到端实跑：Radiohead Creep 下载 0%→100%（3785324 字节）→合并→检测 mp3→ID3 嵌入封面（3.78MB→3.97MB）→保存→`audioExists=true`。
- TS：`write_compat.ts` 零诊断。
- commit `fdb50907`。
